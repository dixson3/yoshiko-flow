# Exp 002 — `_shared/` consumption mechanism (for the renderable-fence registry)

**Question:** how is `_shared/` consumed today, so a new `_shared/renderable_fences.py`
can become the single source of truth for the renderable fence-class registry imported
(effectively) by `markdown_lint.py` and mirrored into the pandoc Lua filter?

## 1. What's in `_shared/` now

| File | Purpose / metadata |
|:-----|:-------------------|
| `active_set.py` | Active-set classifier. Marker-fenced **region** asset. No PEP-723 (library body). |
| `json_extract.py` | Defensive `--json` extractor (`_extract_first_json`). Marker-fenced **region** asset. |
| `manifest_update.py` | Manifest hash/version updater. **Whole-file** asset (copied byte-for-byte). |
| `sync.py` | Repo-time sync engine. PEP-723 header. Defines `REGION_ASSETS` + `WHOLE_FILE_ASSETS` + `--check`. |
| `test_sync.py` | Pytest for the sync engine. |
| `README.md` | Documents the vendoring pattern. |

Only `sync.py` is independently runnable. Canonical library files are sources for
vendoring, not importable packages.

## 2. How skills consume `_shared` today — the mechanism

**There is no runtime import.** `_shared` is consumed at **repo-authoring time** via
committed **vendored copies**. No `sys.path`, no `importlib`, no `[tool.uv.sources]`, no
symlinks. From `_shared/README.md`: skills must be independently installable, so a shared
helper is *vendored as a committed copy inside each consuming script, not imported*.

Two shapes, both regenerated by `uv run _shared/sync.py`:

- **Region** — consumer carries a marker-fenced block; `sync.py` overwrites only the
  interior. Example, `skills/yf-plan/scripts/plan_manager.py`:
  ```
  # >>> BEGIN defensive json extractor (generated by _shared/sync.py — edit _shared/json_extract.py) >>>
  ...vendored body...
  # <<< END defensive json extractor (generated by _shared/sync.py — edit _shared/json_extract.py) <<<
  ```
- **Whole-file** — consumer *is* the shared file (e.g. `manifest_update.py`), copied verbatim.

Registries: `REGION_ASSETS` / `WHOLE_FILE_ASSETS` in `_shared/sync.py`. `sync.py --check`
exits non-zero on divergence (CI/manual backstop). Provenance: plan-016 Epic A chose
"canonical in repo-root `_shared/`, copies are generated artifacts." Install-time deploy
(#41) and PEP-723-source (#40) are explicitly deferred.

## 3. Constraints for a new `renderable_fences.py`

- **Lives at** `_shared/renderable_fences.py` (canonical), a plain module holding the
  registry. **Region** shape fits — `markdown_lint.py` is a large existing script that
  embeds a snippet.
- **`markdown_lint.py` reaches it NOT via import** — it gets a new marker-fenced region
  whose interior is the vendored registry. PEP-723 header stays `dependencies = []`;
  `uv run` keeps working (helper inlined, not imported).
- **Sync registration:** add a `RegionAsset` to `REGION_ASSETS` (canonical →
  `markdown_lint.py`) and extend `test_sync.py`.
- **install.sh: none exists.** Install is the compiled `yf` binary; `yf skills install`
  embeds/deploys the `skills/` tree only. `_shared/` sits outside `skills/` and the
  vendored copy already lives inside `skills/yf-markdown-lint/scripts/markdown_lint.py`,
  so **the install path needs zero changes** — ships zero `yf` Rust changes.
- **DRIFT-CHECK.md:** add a canonical node + copy node, an edge, and §6 trigger globs,
  mirroring the `json-extract` precedent (§1 nodes, §2 edges, §3 contract, §6 globs).

## 4. The Lua-mirror sync problem

The pandoc Lua filter (new, in `yf-markdown-pdf`) hardcodes a fence-class list that must
agree with `_shared/renderable_fences.py`. Lua can't import Python. Options:

- **(a) pytest set-equality** — parse both, assert equal sets. Fires only on test runs.
- **(b) DRIFT-CHECK `field-set-equal` edge** — semantic set-equality across two syntaxes;
  fits the engine (per-edge evidence sub-agent), fires on-edit when either file changes.
- **(c) Generate the Lua list from the Python registry** via a new `sync.py` emitter into
  a marker-fenced region in the `.lua` file — single author, `sync.py --check` becomes the
  byte-level guard, no rot vector.

**Recommendation: (c) backstopped by (b).** (c) is the faithful extension of the repo's
"copies are generated, never hand-maintained" philosophy; (b) is the on-edit enforcement
that catches a hand-edit of the generated Lua region. (a) becomes redundant once
`sync.py --check` exists in `test_sync.py`. If the Python→Lua emitter proves too invasive,
fall back to **(a)+(b)** with the Lua list hand-maintained and policed by the drift edge.

## Net recommendation for the plan

- Canonical `_shared/renderable_fences.py` (region asset) is the registry SoT.
- `markdown_lint.py` vendors a marker-fenced region; register in `sync.py` + `test_sync.py`.
- Lua filter list is **generated** from the registry by a new `sync.py` emitter (option c),
  with a DRIFT-CHECK `field-set-equal` edge (option b) as on-edit enforcement.
- No `yf`/Rust/install changes — `_shared` stays outside `skills/`, copies ship inline.
